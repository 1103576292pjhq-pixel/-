# `llmt_col` 三级流水讲解

## 1. 这个文件在做什么
`rtl/llmt_col.v` 是整个阵列里“单列”的基本计算单元。  
它每次吃进一组 `A` block 和一组 `B` block，先做 `dot32`，再把结果转成 `FP32`，最后和列内累计值相加。

如果把 `mx_array_32x16.v` 看成“16 个并排的工人”，那 `llmt_col.v` 就是每个工人的工作流程模板。

## 2. 输入输出怎么理解
- `valid_i`：这一拍输入是不是有效 block
- `acc_clear_i`：这一拍是不是要从 0 开始累加
- `a_elems_i` / `b_elems_i`：32 个 `E4M3` 元素打包后的 block
- `a_scale_i` / `b_scale_i`：对应 block 的 `E8M0` scale
- `valid_o`：这一列的输出这一拍是不是更新完成
- `acc_o`：当前列内累计得到的 `FP32` 结果

读代码时可以把它想成一句话：  
“拿两个 `MXFP8` block 做一次块级点积，再把结果并到当前列的 `FP32` 累加器里。”

## 3. 三级流水怎么分
当前版本不是最激进的竞赛版，而是一个更容易验证的保守三级流水：

1. `S1`：先把 32 个 lane 切成 4 组，每组各自累加出 partial sum，并把这 4 组 partial sums 连同 `NaN` 标记和指数偏移一起寄存起来。
2. `S2`：把 `S1` 寄存下来的 4 组 partial sums 再做两级合并，得到最终整数点积，然后通过 `fixed_to_fp32` 转成 `FP32`，并把这个 `FP32` 点积结果寄存起来。
3. `S3`：用 `fp32_add_rne` 把上一步的 `FP32` 点积与当前 `acc_reg` 相加，然后写回累加器。

这样切分的直接好处是：  
原来一拍里串在一起的“点积 + 转换 + 累加”被拆开了，关键路径更短，testbench 也能明确根据 `valid_o` 对齐结果。

相比更早的串行写法，现在 `S1` 不但把最重的 `dot32` 部分改成显式分组归约树，还把 final merge 推迟到下一段，关键路径不再是一条从第 0 个 lane 串到第 31 个 lane 的长加法链。

## 4. 代码里最值得看的 always 块
主时序块同时维护三件事：

- 把输入 block 推进到 `S1`
- 把 `S1` 的 `FP32` 点积推进到 `S2`
- 在 `S3` 决定是“直接清零后写首块结果”，还是“在旧累加值上继续相加”

可以重点看下面这几个寄存器：

- `valid_s1` / `valid_s2`：标记流水线里哪一级真的装着有效数据
- `acc_clear_s1` / `acc_clear_s2`：把“这一块是否是新的累加起点”一路带到写回级
- `dot_sum_s1` / `dot_exp_shift_s1`：保存块级点积的整数值和指数偏移
- `dot_fp32_s2`：保存已经转好的 `FP32` 点积
- `acc_reg`：列内当前累加结果

## 5. 为什么 testbench 现在要等 `valid_o`
流水化以后，输入有效不再等于“下一拍就有结果”。  
所以 `tb_llmt_col_smoke.v`、`tb_llmt_col_corner.v`、`tb_mx_array_smoke.v`、`tb_mx_array_dataset.v` 都改成了按 `valid_o` 等结果，而不是写死单拍时序。

这点很重要，因为矩阵级 testbench 需要区分：

- 中间 block 产生的部分和
- 最后一个 block 才得到的最终结果

如果继续按单拍假设检查，就会把“部分和”误当成“最终和”。

## 6. 读这个文件的建议顺序
建议按下面顺序看：

1. 先看组合逻辑里怎么把 32 个 `E4M3` 元素分成 4 组 partial sums
2. 再看 `fixed_to_fp32` 和 `fp32_add_rne` 两个模块的调用位置
3. 最后看主时序块，确认这 4 组 partial sums 怎样从 `S1 -> S2 -> S3`

如果你能自己复述出“一个 block 什么时候进入流水、什么时候变成 `FP32`、什么时候真正写进 `acc_reg`”，这份文件就算读懂了。
